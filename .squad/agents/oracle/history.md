- **[2026-04-21] FONT CONTROL REDESIGN: UX/ACCESSIBILITY REVIEW (Pre-Implementation):** Reviewed proposal to replace current two-select pattern (Filter dropdown + Font dropdown) with complex control combining font selection, filtering, search, and add-font actions (Docs/Word model). Findings: (1) **Concept Sound:** Combining functions in one control matches target audience mental model and competitors; no accessibility barriers identified if implemented correctly. (2) **Keyboard Navigation Required:** Must implement custom combobox-like behavior with arrow keys (open/navigate), Tab (flow), Escape (close), Enter (select), Home/End (jump), type-ahead (search). Cannot use native `<select>` element; requires manual focus management (roving tabindex pattern). (3) **Screen Reader Semantics:** Use `role="listbox"` on dropdown container, `role="button" aria-expanded="true|false" aria-haspopup="listbox"` on trigger, `role="option"` on font items. Announce search results count via `aria-live="polite"`. (4) **Selection-vs-Insertion Model:** Current code applies font to selection only; Docs/Word also apply pending font to insertion point (next text typed). Recommend adding `pendingFontFamily` state in TextEditor.tsx to track this and display in trigger button with aria-label ("Will apply to new text"). (5) **Implementation Complexity:** Doable with SolidJS; focus management skeleton provided in decision. No blockers, but requires careful testing (keyboard nav, screen reader, mobile touch model TBD). **Key Accessibility Patterns:** Manual focus management with roving tabindex, ARIA live regions for search feedback, clear exit paths (Escape always closes + returns focus to trigger). **WCAG 2.2 AA Compliance:** All checkpoints addressable (focus visible, keyboard access, aria attributes, status messages). Decision document: `.squad/decisions/inbox/oracle-font-control-review.md`. Guidance complete; ready for Trinity implementation.
